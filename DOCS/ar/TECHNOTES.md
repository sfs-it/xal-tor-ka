# Xal-Tor-Ka — ملاحظات تقنية (ما الذي يفعله وكيف)

*ترجمة للوثيقة الإنجليزية الرسمية؛ في حال وجود أي اختلاف، تكون النسخة الإنجليزية هي المرجع.*

شرح تقني لكنه سهل القراءة لكيفية عمله. للاطّلاع على **المواصفة
الدقيقة** (مخطّط JSON، نماذج البيانات، عقد نقاط النهاية) انظر
[`BLUEPRINT.md`](../../BLUEPRINT.md)؛ ولتثبيته، انظر [`REQUIREMENTS.md`](../../REQUIREMENTS.md)
+ [`INSTALL.md`](../../INSTALL.md).

## في جملة واحدة

إنّ Xal-Tor-Ka هو **بوّابة مصادقة + مدير وكيل عكسي (reverse-proxy)**: يضع
**NGINX** بوصفه الخدمة الوحيدة المعرَّضة للإنترنت، ويسأل لكل طلب
**خدمة Go داخلية** (غير معرَّضة أبداً) عمّا إذا كان يجب السماح بالمرور، أو طلب تسجيل الدخول، أو الرفض.

```
Internet → NGINX (gatekeeper) ──auth_request (internal)──► Xal-Tor-Ka (Go)
                  │ 200 = pass / 401 = login / 403 = denied
                  └── proxy_pass (only if authorized) ──► internal backends
```

## كيف يقرّر: تدفّق `auth_request`

إنّ NGINX **لا يعرف القواعد**. فلكل طلب وارد يُجري طلباً فرعياً
*subrequest* داخلياً إلى نقطة النهاية `/validate` في خدمة Go، ممرِّراً المضيف الأصلي
والمسار. وتردّ الخدمة برمز حالة HTTP:

- **200** → يتابع NGINX بـ `proxy_pass` إلى الواجهة الخلفية الحقيقية؛
- **401** → يعيد NGINX التوجيه إلى صفحة تسجيل الدخول؛
- **403** → الوصول مرفوض.

المبدأ هو **fail-closed**: أي خطأ أو مهلة منتهية أو شكّ أثناء التقييم
يؤدّي إلى 401/403، و**لا يؤدّي أبداً** إلى 200. خدمة Go غير قابلة للوصول من الخارج: فقط
NGINX، على الشبكة الداخلية، يمكنه الاستعلام منها.

## النواة: مصفوفة التفويض

لكل **مضيف** و**مسار**، تنطبق إحدى ثلاث قواعد:

| القاعدة | مَن يدخل |
|------|-------------|
| `public` | أي شخص |
| `authenticated` | المستخدمون ذوو الجلسة الصالحة مع إتمام **2FA** (TOTP) |
| `whitelist` | المستخدمون **المصرَّح لهم صراحةً** لتلك الخدمة فقط |

يمكن **للمسؤولين** الوصول إلى كل شيء. الدقّة تكون لكل نطاق فرعي ولكل مسار
(مثلاً `/` عام لكن `/admin` ضمن whitelist على المضيف نفسه).

## المصادقة

- **محلية (Local)**: كلمات المرور مُجزّأة (hashed) بـ **argon2id** + عامل ثانٍ عبر **TOTP**
  (RFC 6238، متوافق مع Google Authenticator/Authy).
- **OIDC** (OpenID Connect): تسجيل الدخول مفوَّض إلى **Google** أو **Microsoft/Entra** أو
  مزوّدين عامّين (**Keycloak, Authentik, Auth0, Okta, GitLab**). يجري التحقّق من توقيع
  رمز الهوية مقابل المفاتيح العامة للمزوّد.
  - **بدون توفير تلقائي (auto-provisioning)**: يجب أن يكون المستخدم موجوداً مسبقاً،
    معلَناً لذلك المزوّد — لا يكفي تسجيل الدخول بـ Google للدخول. انظر
    [`AUTH-PROVIDERS.md`](../../AUTH-PROVIDERS.md).
- **الجلسات**: ملفّات تعريف الارتباط (cookies) بخصائص `HttpOnly`/`SameSite=Lax` (و`Secure` خلف HTTPS)،
  محفوظة في ذاكرة RAM مع ثبات على الملفات (تبقى بعد إعادة التشغيل).

## الإعداد: كل شيء في JSON

لا حاجة لقاعدة بيانات إلزامية: يعيش الإعداد في بضعة ملفات JSON مُنمّطة، يجري التحقّق منها
عند بدء التشغيل (**Fail-Fast**: أي حقل غير معروف أو قيمة خارج النطاق يمنع بدء التشغيل برسالة
واضحة).

| الملف | المحتوى |
|------|---------|
| `config.json` | البنية التحتية (بقوالب من متغيّرات البيئة): وضع المصادقة، TLS، الجلسات، عناوين IP للمسؤول، المزوّدون |
| `secrets.json` | الأسرار (أسرار عميل OIDC، الرموز، SMTP) — لا تُوضع تحت إدارة الإصدارات أبداً |
| `users.json` | المستخدمون، الأدوار، 2FA، التفويضات — لا تُوضع تحت إدارة الإصدارات أبداً |
| `services.json` | الخدمات المُدارة وقت التشغيل (الواجهات الخلفية عبر الوكيل + روابط لوحة التحكّم) |

تنطبق التغييرات على المستخدمين/الخدمات **فوريّاً** (hot reload)، دون إعادة تشغيل.

## المكوّنات (خدمة Go)

تعتمد على المكتبة القياسية أولاً (Stdlib-first)، بملفّ ثنائي ساكن (static). الحِزَم الرئيسية:

- `handlers/` — نقاط نهاية HTTP: `/validate`، تسجيل الدخول + TOTP، ردّ نداء OIDC، الإعداد،
  لوحة `/admin`، ولوحة تحكّم `/listing`.
- `providers/` — المصادقة: `local` و`oidc` (واجهة مشتركة).
- `matrix/` — تقييم قواعد التفويض (لكل مضيف/مسار).
- `proxy/` — يولّد المضيفات الافتراضية (vhosts) للواجهة الخلفية في NGINX ويعيد التحميل.
- `health/` — فحوصات صحّة دورية للواجهات الخلفية (نقطة النهاية `/health`).
- `config/` — التحميل + التحقّق + الحفظ الذرّي مع اللقطات (snapshots).
- `audit/` — سجلّ لمحاولات الوصول الفاشلة (لأجل fail2ban).
- `auth/` — التجزئة (hashing)، TOTP، الجلسات، دليل المستخدمين.

## الوكيل العكسي: التوليد وإعادة التحميل

يولّد المدير إعداد الواجهة الخلفية لـ NGINX (كتلة `server{}` واحدة لكل مضيف،
مع `auth_request` على المسارات المحميّة و`proxy_pass` إلى الوجهة العليا upstream). أمّا
**إعادة التحميل** فهي:

- **Docker**: حاوية NGINX تكتشف التغيير وتعيد تحميل نفسها (عبر الاستطلاع polling)،
  لأنّ `inotify` غير موثوق على نقاط الربط (bind mounts) في Docker Desktop/WSL2.
- **Host/LXD**: تشغّل خدمة Go أمر إعادة تحميل قابلاً للتهيئة
  (`nginx -s reload` / `systemctl reload nginx`).

يتحقّق NGINX دائماً من الإعداد الجديد، وإذا كان غير صالح يُبقي الإعداد العامل
الحالي: فالتوليد الخاطئ لا يُسقط الوكيل.

## الإدارة والتشغيل

- **لوحة `/admin`** (مقيَّدة بعناوين IP): إدارة الخدمات والمستخدمين والصلاحيات،
  ومراقبة الحالة، في صفحات منفصلة.
- **لوحة تحكّم `/listing`**: تُظهر لكل مستخدم الخدمات التي يمكنه الوصول إليها فقط.
- **التهيئة الأولية (Onboarding)**: يولّد التشغيل الأول رمزاً منتهي الصلاحية لإنشاء أول
  مسؤول عبر الويب؛ ثم تُحصَّن الواجهة.
- **النسخ الاحتياطية**: كل عملية حفظ تنشئ لقطة (snapshot) مع حذف تلقائي (auto-trash) (يُبقي آخر N)،
  مع إمكانية الاستعادة من سطر الأوامر (CLI) أيضاً.
- **الدفاع ضد التخمين القسري**: تنتهي المحاولات الفاشلة في سجلّ مهيكل
  (`logs/auth.log`) مع عنوان IP الحقيقي للعميل، قابل للدمج مع **fail2ban**.

## الأمان في لمحة

- الخدمة الوحيدة المعرَّضة هي **NGINX**؛ وخدمة Go داخلية.
- مبدأ **Fail-closed** عبر مسار التفويض بأكمله.
- مقارنات الأسرار تتمّ في **زمن ثابت (constant time)**؛ والأسرار **لا تُسجَّل أبداً**.
- منطقة المسؤول مقيَّدة بـ **عنوان IP**؛ ويؤخذ عنوان IP الحقيقي للعميل من
  `X-Forwarded-For` فقط من الوكلاء الموثوقين.
- رمز الإعداد **يُستخدم لمرة واحدة (single-use)** ومنتهي الصلاحية.

## النشر (Deployment)

- **Docker Compose** (الافتراضي): NGINX معرَّض، وخدمة Go داخلية، وحاوية جانبية (sidecar)
  للقراءة فقط لاكتشاف الحاويات، مع حدود للموارد وتدوير للسجلّات.
- **Host / LXD / جهاز مخصّص**: ملفّ ثنائي ساكن + NGINX النظامي، محكوم بثلاثة
  متغيّرات (`DEPLOY_MODE`، `NGINX_RELOAD_CMD`، `UPSTREAM_LOCALHOST`).
  انظر [`INSTALL.md`](../../INSTALL.md) §9.
