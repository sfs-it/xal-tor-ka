# Xal-Tor-Ka — Технические заметки (что это делает и как)

*Перевод официального англоязычного документа; при любых расхождениях приоритет имеет английская версия.*

Технически, но доступно изложенное объяснение принципа работы. За **строгой
спецификацией** (схема JSON, модели данных, контракт эндпоинтов) обращайтесь к
[`BLUEPRINT.md`](../../BLUEPRINT.md); для установки смотрите [`REQUIREMENTS.md`](../../REQUIREMENTS.md)
+ [`INSTALL.md`](../../INSTALL.md).

## В одном предложении

Xal-Tor-Ka — это **шлюз аутентификации + менеджер обратного прокси**: он ставит
**NGINX** единственным сервисом, открытым в интернет, и для каждого запроса
спрашивает **внутренний сервис Go** (никогда не открытый наружу), пропустить ли,
запросить ли вход или отказать.

```
Internet → NGINX (gatekeeper) ──auth_request (internal)──► Xal-Tor-Ka (Go)
                  │ 200 = pass / 401 = login / 403 = denied
                  └── proxy_pass (only if authorized) ──► internal backends
```

## Как он принимает решение: поток `auth_request`

NGINX **не знает правил**. Для каждого входящего запроса он делает внутренний
*подзапрос* к эндпоинту `/validate` сервиса Go, передавая исходные хост и путь.
Сервис отвечает HTTP-статусом:

- **200** → NGINX продолжает с `proxy_pass` к реальному бэкенду;
- **401** → NGINX перенаправляет на страницу входа;
- **403** → доступ запрещён.

Принцип — **fail-closed** (отказ в безопасную сторону): любая ошибка, тайм-аут
или сомнение при оценке даёт 401/403, **никогда** 200. Сервис Go недоступен
извне: запрашивать его может только NGINX во внутренней сети.

## Ядро: матрица авторизации

Для каждого **хоста** и **пути** применяется одно из трёх правил:

| Правило | Кто получает доступ |
|---------|---------------------|
| `public` | кто угодно |
| `authenticated` | пользователи с действующей сессией и пройденной **2FA** (TOTP) |
| `whitelist` | только пользователи, **явно авторизованные** для этого сервиса |

**Администраторы** имеют доступ ко всему. Гранулярность — на уровне поддомена и
пути (например, `/` публичный, но `/admin` по белому списку на том же хосте).

## Аутентификация

- **Локальная**: пароли хешируются с помощью **argon2id** + второй фактор через
  **TOTP** (RFC 6238, совместим с Google Authenticator/Authy).
- **OIDC** (OpenID Connect): вход делегируется **Google**, **Microsoft/Entra** или
  обобщённым провайдерам (**Keycloak, Authentik, Auth0, Okta, GitLab**). Подпись
  токена идентичности проверяется по публичным ключам провайдера.
  - **Без автоматического создания учётных записей**: пользователь должен уже
    существовать и быть объявлен для данного провайдера — одного входа через
    Google недостаточно, чтобы попасть внутрь. См.
    [`AUTH-PROVIDERS.md`](../../AUTH-PROVIDERS.md).
- **Сессии**: cookie `HttpOnly`/`SameSite=Lax` (и `Secure` за HTTPS), хранятся в
  RAM с сохранением в файл (переживают перезапуск).

## Конфигурация: всё в JSON

Обязательной базы данных нет: конфигурация живёт в нескольких типизированных
JSON-файлах, проверяемых при запуске (**Fail-Fast**: неизвестное поле или
значение вне диапазона блокирует запуск с понятным сообщением).

| Файл | Содержание |
|------|------------|
| `config.json` | инфраструктура (с шаблонами env): режим аутентификации, TLS, сессии, IP администраторов, провайдеры |
| `secrets.json` | секреты (клиентские секреты OIDC, токены, SMTP) — никогда не версионируется |
| `users.json` | пользователи, роли, 2FA, авторизации — никогда не версионируется |
| `services.json` | управляемые во время работы сервисы (проксируемые бэкенды + ссылки дашборда) |

Изменения пользователей/сервисов применяются **на лету** (hot reload), без
перезапуска.

## Компоненты (сервис Go)

Stdlib-first, статический бинарник. Основные пакеты:

- `handlers/` — HTTP-эндпоинты: `/validate`, вход + TOTP, OIDC callback, setup,
  панель `/admin`, дашборд `/listing`.
- `providers/` — аутентификация: `local` и `oidc` (общий интерфейс).
- `matrix/` — оценка правил авторизации (по хосту/пути).
- `proxy/` — генерирует виртуальные хосты бэкендов NGINX и выполняет перезагрузку.
- `health/` — периодические проверки состояния бэкендов (эндпоинт `/health`).
- `config/` — загрузка + проверка + атомарное сохранение со снимками.
- `audit/` — журнал неудачных попыток доступа (для fail2ban).
- `auth/` — хеширование, TOTP, сессии, каталог пользователей.

## Обратный прокси: генерация и перезагрузка

Менеджер генерирует конфигурацию бэкендов NGINX (один `server{}` на хост, с
`auth_request` на защищённых маршрутах и `proxy_pass` к upstream).
**Перезагрузка**:

- **Docker**: контейнер NGINX обнаруживает изменение и перезагружается сам
  (опрос/polling), потому что `inotify` ненадёжен на bind-монтированиях Docker
  Desktop/WSL2.
- **Host/LXD**: сервис Go выполняет настраиваемую команду перезагрузки
  (`nginx -s reload` / `systemctl reload nginx`).

NGINX всегда проверяет новую конфигурацию и, если она недействительна, сохраняет
работающую: неудачная регенерация не выводит прокси из строя.

## Управление и эксплуатация

- **Панель `/admin`** (с ограничением по IP): управление сервисами,
  пользователями и правами, мониторинг состояния, на отдельных страницах.
- **Дашборд `/listing`**: показывает каждому пользователю только те сервисы, к
  которым у него есть доступ.
- **Онбординг**: первый запуск генерирует токен с истечением срока для создания
  первого администратора через веб; затем интерфейс блокируется.
- **Резервные копии**: каждое сохранение создаёт снимок с auto-trash (хранит
  последние N), с восстановлением в том числе из CLI.
- **Защита от подбора паролей**: неудачные попытки попадают в структурированный
  журнал (`logs/auth.log`) с реальным IP клиента, подключаемый к **fail2ban**.

## Безопасность с одного взгляда

- Единственный открытый сервис — **NGINX**; сервис Go внутренний.
- **Fail-closed** по всему пути авторизации.
- Сравнение секретов в **постоянном времени** (constant time); секреты
  **никогда** не логируются.
- Область администрирования ограничена по **IP**; реальный IP клиента берётся из
  `X-Forwarded-For` только от доверенных прокси.
- Токен установки **одноразовый** и с истекающим сроком.

## Развёртывание

- **Docker Compose** (по умолчанию): NGINX открыт наружу, сервис Go внутренний,
  read-only sidecar для обнаружения контейнеров, лимиты ресурсов и ротация
  логов.
- **Host / LXD / выделенная машина**: статический бинарник + системный NGINX,
  управляемый тремя переменными (`DEPLOY_MODE`, `NGINX_RELOAD_CMD`,
  `UPSTREAM_LOCALHOST`). См. [`INSTALL.md`](../../INSTALL.md) §9.

## Версия

Единый источник истины в `version/version.go` (линия до 1.0 `beta0.N`),
переопределяемый во время сборки, и отображается в `xaltorka version`, `/healthz`,
журнале запуска и интерфейсе.
